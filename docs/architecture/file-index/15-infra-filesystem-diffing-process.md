## Subsystem role
These three infrastructure subsystems are the low-level substrate that the rest of the app builds on. **FileSystem** owns the workspace root abstraction: an `actor FileSystemService` (with a fleet of same-target extensions) that enumerates directories via POSIX `opendir`, loads and encoding-detects file content, watches for changes through FSEvents (ingested through a synchronous deep-copying mailbox), performs sandbox-guarded file mutations, and applies git-style hierarchical ignore rules through a compiled-pattern stack backed by LRU/path/pattern caches. **Diffing** is the edit-flow engine: it parses LLM-emitted search/replace and diff blocks, generates `DiffChunk`s via Myers diff with indentation correction and optimal-line matching, applies them to file content with positional bookkeeping, and emits unified diffs — all instrumented by the gated `EditFlowPerf` telemetry harness. **Process** is the child-process / CLI launch substrate: it spawns processes via `posix_spawn`, builds and sanitizes child environments (DYLD scrubbing, login-shell capture, PATH defaults), resolves command paths via the user's interactive shell, frames NDJSON stream output, handles graceful/forced termination, and installs the CLI tool into the user's PATH via managed symlinks and a `claude-rp` wrapper.
